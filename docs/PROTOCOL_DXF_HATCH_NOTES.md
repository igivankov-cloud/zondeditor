# Protocol DXF Hatch Notes

## Зачем нужен этот файл
- Это памятка по самой болезненной части экспорта DXF-протокола: штриховкам разреза.
- Цель: не возвращаться к циклу из локальных хаков по знакам, origin и отдельным паттернам.

## Источники истины
- Источник формы паттерна: PAT-файлы в `src/zondeditor/assets/hatches`.
- Источник DXF-семантики `HATCH`: эталонные DXF, сохранённые самим AutoCAD, в `shtrihovkidxf`.
- Если визуально что-то снова ломается, сравнивать нужно не только PAT-строки, а полную `HATCH` entity по group codes.

## Что в итоге оказалось критичным
- Совпадение raw PAT rows с содержимым DXF ещё не гарантирует правильную картинку в AutoCAD.
- Главная ошибка была в `group codes 45/46` для наклонных строк.
- В PAT-файле offset хранится в локальной системе строки.
- В DXF `HATCH`, который пишет AutoCAD, `45/46` для custom pattern уже лежат в мировых `X/Y`.
- Поэтому горизонтальные/вертикальные семейства могли выглядеть правильно, а наклонные распадались.

## Правильная семантика для DXF protocol hatch
- Имя паттерна должно быть реальным именем PAT (`graviy`, `peschanik`, `argillit`, ...), а не внутренним alias вида `ZE_*`.
- `pattern_type` для этих штриховок должен быть `2` (`custom`).
- `angle`, `base_point`, `dash/gap` берутся из PAT.
- `offset` для DXF `45/46` нужно переводить из локального PAT-step в мировые координаты по углу строки.
- Для сравнения использовать эталонные DXF из `shtrihovkidxf`, а не только ручные формулы.

## Формула перевода PAT offset -> DXF 45/46
Для PAT строки:
- угол `a`
- локальный offset `(dx_local, dy_local)`

мировой offset для DXF:

```text
ex = (cos(a), sin(a))
ey = (-sin(a), cos(a))

dx_world = dx_local * ex.x + dy_local * ey.x
dy_world = dx_local * ex.y + dy_local * ey.y
```

Именно `dx_world/dy_world` должны попадать в `45/46`.

## Эталонные примеры из AutoCAD
- `graviy` 45°: `45/46 = -4.5 / 4.5`
- `graviy` 135°: `45/46 = -4.5 / -4.5`
- `argillit` 135°: `45/46 = -5.999942 / -5.999942`
- `peschanik` 60°: `45/46 = -0.000002 / 11.999996`
- `peschanik` 120°: `45/46 = -10.392302 / 5.999996`

Если в будущем эти значения снова превращаются в PAT-local offsets вроде `0 / 6.36`, значит DXF-конвертация сломана.

## Что сравнивать при новой диагностике
Минимальный чек-лист по `HATCH`:
- `41 / 47 / 52`
- `70 / 71 / 75 / 76 / 77`
- `91 / 92 / 93 / 97 / 98`
- `53 / 43 / 44 / 45 / 46 / 49 / 79`
- boundary loop и порядок кодов

Но сначала смотреть именно на `45/46` для наклонных строк: это был корневой дефект.

## Практическое правило на будущее
- Не чинить сложные штриховки “на глаз” по одному паттерну.
- Не делать новые знаковые хаки без сравнения с AutoCAD DXF-эталоном.
- При любом изменении этого участка прогонять сравнение с `shtrihovkidxf`.
